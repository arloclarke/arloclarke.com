When the backlog runs out

A framework for autonomous product progress

It is now easy to point AI agents at a software project. It is still hard to make their work compound into product progress. Without a clear operating loop, agents default to task churn: branches, artifacts, and plausible-looking direction that do not move the product closer to what it exists to do. Much of the value of an autonomous agent comes from its ability to keep moving when the roadmap is incomplete. If it requires a fully defined backlog to operate usefully, much of that value is lost. Here I’ll outline a framework for agentic product progress: autonomous software development that makes progress toward product goals with minimal human input. The framework has five parts: an objective function, a runnable product spine, an ordered backlog, a pressure loop for when the backlog is empty, and a human escalation path for ambiguity.

While the described framework isn't incompatible with multi-agent workflows, I'll assume a single code-writing agent for the rest of this note to avoid discussing the challenges of concurrency.

Define an objective function

The single most important step in achieving autonomous product progress is defining a product goal precise enough to guide decisions. The goal does not need to be fully numeric, but it must make distance legible: the agent should be able to tell whether a change moved the system closer to or farther from the objective. Without that signal, small autonomous steps quickly become arbitrary.

Bad example of an objective function:

Project goal is to build a system to autonomously drive a passenger vehicle.

Slightly better example of an objective function:

Project goal is to build a dashboard-mounted, single-camera system for autonomously driving a passenger vehicle. Given a 60 fps RGB video input, the core stack should output vehicle control directives such that the passenger arrives at a predefined GPS destination in minimum time while obeying traffic laws and, most critically, protecting the safety of the passenger and surrounding entities.

If the bad example is supplied to an agentic loop, it will likely spend weeks or months building systems and architecture around unhelpful requirements; for example, the agent may choose to invest deeply in a LIDAR stack. The slightly better example is well specified and allows an agent to measure progress toward the goal against what it’s built so far.

Define a product spine

For useful autonomous progress, we first need to define a product spine such that the primary operational model, with inputs and outputs, can be run, rerun, and evaluated. Without a defined spine, the agent will create diverging and often conflicting product logic depending on isolated task goals. A defined product spine forces all tasks to operate coherently within useful boundaries. A well-defined objective function should allow a product spine to follow intuitively. For the slightly better objective function example above, a product spine would take a user’s starting and destination locations, then iteratively take a new streamed video frame, output a control directive, execute that directive in the simulated environment, and accept the new video frame generated by the environment.

Define a backlog and a product pressure loop

A backlog can contain one-off improvements, features, or even an implementation roadmap. This is helpful when you, as the human director, have insight that can course-correct the agent’s progress. An agent should immediately work on backlog items in order when any exist. If no backlog items exist, the agent should fall back to its default operational mode: the product pressure loop. The pressure loop is the primary method by which autonomous progress is made. The product pressure loop should choose tasks that:

  • Address the highest-leverage gap needed to operate the product spine appropriately
  • Apply adversarial product pressure to the spine by choosing operating inputs such that the product spine is likely to fail or move further away from the objective function
  • Build generalized fixes and improvements to the spine such that any failures identified through adversarial pressure are addressed and the spine moves closer to the objective function

This does not guarantee the end goal will be achieved given enough time, but it makes agentic build time less likely to dissolve into churn because each task is selected against the same spine and objective.

Implement periodic cleanup, hardening, and refactors

An unfortunate side effect of greedy, incremental progress toward an end goal is that isolated task implementations can produce patchwork solutions when a more holistic architectural approach may be more appropriate. To account for that, the agent should periodically run a review process to identify candidates for high-leverage refactoring and cleanup. For any high-leverage items identified, the agent should implement them immediately before resuming the backlog or product pressure loop.

Implement ambiguity recognition and handling

Finally, we need to implement a method for identifying ambiguity in a plan, in the path toward the objective, or in the objective itself. Handling ambiguity poorly risks poor technical or product decision-making and wastes time and tokens. The agent control plane should encode clear guidance on what to do when encountering ambiguity or a blocker with multiple reasonable paths forward. Almost always, the best way to handle ambiguity is to ask for human input by whatever means is convenient.

Toy Example: a route-planner pressure loop

The above framework is pretty abstract, so a toy example helps make concrete. Implemented for illustration is a route planner pressure loop.

Toy route-planner objective function:

Given a messy request, return route guidance that is useful enough for an operator to act on or, when the request is underspecified, ask for the right clarification.

Its product spine is narrow: a single plan_route(start, destination) function that can be run again and again against route cases. Once the seed backlog is empty, the question is no longer “what task did a human write down next?” It is “where does this spine fail under pressure?”

The pressure loop answers that by making the planner operate. It generates adversarial cases from the failures it has already seen, proposes replacement implementations, executes every candidate against the same active case bank, scores the observed outputs, and accepts only the candidate that improves the spine. The agent is not rewarded for producing a plausible diff. It is rewarded for moving the runnable product closer to the objective.

candidates = propose_candidate_repairs(
    current_source, case_rows, version, accepted_repairs
)

scored_candidates = []
for candidate in candidates:
    candidate_path = write_candidate(
        candidate["source"], version + 1, candidate["ordinal"]
    )
    candidate_score, candidate_rows = evaluate(
        candidate_path, version + 1, active_cases
    )
    scored_candidates.append((candidate_score, candidate, candidate_rows))

best_score, best, best_rows = max(scored_candidates, key=lambda row: row[0])
if best_score <= score + 0.0001:
    break

version += 1
file = write_snapshot(version, best["label"], best["source"])
record_trace(file, best_rows, best_score)

The cases are adversarial in product terms, not just random unit tests. A weak planner can look fine on “Boston to Cambridge” and still fail the work that matters: coordinate tuples, coordinate strings, latitude/longitude dictionaries, named places, aliases, fuzzy place strings, missing destinations, invalid origins, ambiguous out-of-area requests, reverse legs, and multi-stop routes. As one class gets handled, pressure moves to the remaining gaps: clarification behavior, summaries, safety notes, alternatives, leg quality, normalized request audits, route-mode classification, confidence calibration, and handoff readiness.

The scoring is only a comparison signal, but that is enough for the loop. Each candidate has to execute before it can be scored. The pressure bank can change between accepted versions, so adjacent averages are not a fixed benchmark. The fair comparison happens inside each iteration, where candidates face the same cases and the loop keeps the best improvement.

The toy route planner uses real geography tools and open-ended candidate snapshots. Each snapshot executes against an instrumented Nominatim/OpenStreetMap geocoder and OSRM router, and the score is calibrated from execution evidence: missing features, missing tools, route counts, status matches, and maturity signals. In the checked run, the loop accepted 50 snapshots across 42 real-geography pressure cases, moving the evidence-calibrated mean score from 0.035 to 0.925 and the final worst case to 0.800.

What matters in the run is not the average rising in isolation. The early versions learn direct parsing and route production; later versions add bounded region checks, multi-leg routing, nearby-POI search, clarification packets, confidence notes, and operator-facing handoff details. The worst-case line only rises after those remaining cases are actually covered. That is the pressure loop doing its job: not passing yesterday’s examples forever, but using yesterday’s failures to manufacture tomorrow’s useful work.

This is the route-planner version of the framework above. The objective makes distance legible, the spine makes progress executable, the empty backlog hands control to pressure, and the audit trail keeps the work honest. Human input is still available for ambiguity, but the default motion is no longer arbitrary task churn; it is repeated pressure against the same product spine until the product becomes more capable.

Wrap Up

When all these framework pieces work in concert, real, rapid progress toward an unmapped product end goal can be achieved with minimal human input.
